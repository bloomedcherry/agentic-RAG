## 1. 评测目标

本评测方案用于验证企业 Agentic RAG Demo 是否真正提升了以下能力：

1. 企业知识检索是否准确；
2. 工具调用是否正确；
3. 回答是否有依据；
4. 多步任务是否能完成；
5. Verifier / Retry 是否能减少无依据回答和失败任务；
6. Permission / Safety 是否能拦截越权请求；
7. Trace Logging 是否能支撑错误分析和项目复盘；
8. Harness Runtime 改造是否相对普通 RAG 有可量化收益。

当前评测不评价模型训练效果，不涉及 SFT、DPO、PPO 或 Agentic RL。

评测对象是：

**单 Supervisor Agent + RAG + Tool Use + Harness Runtime**

不是复杂 Multi-Agent 系统。

---

## 2. 评测版本设计

为了证明 Harness Engineering 的收益，建议设计 4 个版本做消融对比。

| 版本 | 名称                        | 组成                                      | 目的              |
| -- | ------------------------- | --------------------------------------- | --------------- |
| V0 | Baseline RAG              | 仅 RAG 检索 + LLM 生成                       | 验证普通 RAG 基线     |
| V1 | RAG + Tool Use            | RAG + SQL / Workflow / Report Tool      | 验证工具调用收益        |
| V2 | RAG + Tool Use + Verifier | 在 V1 上加入引用、工具和格式校验                      | 验证校验机制收益        |
| V3 | Full Harness Runtime      | V2 + Permission + Retry + Trace Logging | 验证完整 Harness 收益 |

推荐最终简历和报告主要展示：

```text
V0 → V3 的整体提升
V1 → V3 的 Harness 改造收益
```

注意：如果时间不够，至少保留：

```text
V0：Baseline RAG
V3：Full Harness Runtime
```

## 4. 核心评测指标

## 4.1 RAG Recall@K

### 指标用途

衡量 RAG 是否能把正确文档召回到 top-k 检索结果中。

### 适用任务

* 企业制度问答；
* 流程判断；
* 项目资料分析；
* 合同 / 制度条款检索；
* 报告生成中的知识检索部分。

### 计算公式

```text
RAG Recall@K = top-k 检索结果中命中 gold_docs 的样本数 / 有 gold_docs 标注的样本数
```

### 示例

如果 30 条需要检索的任务中，有 24 条的正确文档出现在 top-5 检索结果中：

```text
Recall@5 = 24 / 30 = 80%
```

### 评测脚本

```text
eval/eval_rag.py
```

---

## 4.2 Citation Accuracy

### 指标用途

衡量最终回答中的引用是否能支撑答案结论。

### 适用任务

* 所有需要 RAG 证据的任务。

### 计算公式

```text
Citation Accuracy = 引用能支撑答案的样本数 / 带引用答案样本数
```

### 判定方式

第一版可以采用规则 + 人工检查：

1. 答案中是否包含 source 或 chunk_id；
2. source 是否来自 retrieved_docs；
3. 回答中的关键结论是否能在引用片段中找到依据。

后续可以加入 LLM-as-Judge。

### 评测脚本

```text
eval/eval_task.py
```

---

## 4.3 Tool Call Accuracy

### 指标用途

衡量 Agent 是否选择了正确工具。

### 适用任务

* 流程判断；
* 数据分析；
* 报告生成；
* 需要 SQL 或 workflow_check 的任务。

### 计算公式

```text
Tool Call Accuracy = 工具选择正确的样本数 / 需要工具调用的样本数
```

### 示例

如果 25 条任务需要调用工具，其中 21 条调用了正确工具：

```text
Tool Call Accuracy = 21 / 25 = 84%
```

### 评测脚本

```text
eval/eval_tool.py
```

---

## 4.4 Tool Success Rate

### 指标用途

衡量工具调用是否成功执行。

### 适用任务

* 所有实际调用工具的任务。

### 计算公式

```text
Tool Success Rate = 工具成功执行次数 / 工具调用总次数
```

### 工具成功标准

| 工具              | 成功标准                  |
| --------------- | --------------------- |
| search_kb       | 返回非空 retrieved_docs   |
| query_sql       | SQL 或受控查询执行成功，返回结构化结果 |
| workflow_check  | 返回 decision 和 reason  |
| generate_report | 返回非空 markdown_report  |
| parse_doc       | 成功解析并返回文本或 chunks     |

---

## 4.5 Task Success Rate

### 指标用途

衡量端到端任务是否完成。

这是最重要的综合指标。

### 计算公式

```text
Task Success Rate = 成功完成任务数 / 总任务数
```

### 成功判定

一个任务成功通常需要满足：

1. task_type 判断正确；
2. 必要工具调用正确；
3. 工具执行成功；
4. 回答非空；
5. 需要引用的任务包含有效引用；
6. 没有 permission violation；
7. Verifier 通过或 retry 后通过。

### 评测脚本

```text
eval/eval_task.py
```

---

## 4.6 Verifier Pass Rate

### 指标用途

衡量系统输出有多少能通过 Verifier 校验。

### 计算公式

```text
Verifier Pass Rate = Verifier 通过样本数 / 总样本数
```

### 注意

Verifier Pass Rate 并不是越高越好。

如果 Verifier 过松，可能所有答案都通过，但并不能保证质量。
因此需要同时看：

```text
Task Success Rate
Citation Accuracy
Error Type Distribution
```

---

## 4.7 Retry Success Rate

### 指标用途

衡量 retry 机制是否能修复失败任务。

### 计算公式

```text
Retry Success Rate = retry 后成功的任务数 / 首次校验失败并触发 retry 的任务数
```

### 适用失败类型

* missing_citation；
* retrieval_empty；
* sql_error；
* format_error；
* ungrounded_answer。

---

## 4.8 Permission Blocking Accuracy

### 指标用途

衡量系统是否能正确拦截越权工具调用。

### 计算公式

```text
Permission Blocking Accuracy = 正确拦截越权请求数 / 应拦截越权请求数
```

### 示例任务

```text
普通员工查询所有部门采购金额统计
普通员工导出全员报销记录
普通员工查询管理层项目预算
```

期望结果：

```text
permission_denied
```

---

## 4.9 Average Latency

### 指标用途

衡量系统平均响应时间。

### 计算公式

```text
Average Latency = 所有任务耗时总和 / 总任务数
```

### 进一步拆分

可以统计：

| 延迟类型              | 含义       |
| ----------------- | -------- |
| retrieval_latency | RAG 检索耗时 |
| tool_latency      | 工具调用耗时   |
| llm_latency       | LLM 生成耗时 |
| verifier_latency  | 校验耗时     |
| total_latency     | 总耗时      |

---

## 4.10 Error Type Distribution

### 指标用途

统计失败原因，用于分析系统短板。

### 错误类型

| error_type           | 含义           |
| -------------------- | ------------ |
| retrieval_failure    | 检索失败         |
| tool_error           | 工具执行失败       |
| wrong_tool           | 工具选择错误       |
| missing_citation     | 缺少引用         |
| permission_denied    | 权限不足         |
| format_error         | 输出格式错误       |
| verifier_failed      | Verifier 未通过 |
| unsupported_task     | 当前系统不支持任务    |
| llm_generation_error | LLM 生成失败     |

### 评测脚本

```text
eval/analyze_trace.py
```

## 6. 消融实验设计

## 6.1 消融版本

建议设置：

| 版本 | 功能组成                                    |
| -- | --------------------------------------- |
| V0 | Baseline RAG                            |
| V1 | Baseline RAG + Tool Use                 |
| V2 | V1 + Verifier                           |
| V3 | V2 + Retry + Permission + Trace Logging |

### V0：Baseline RAG

流程：

```text
用户问题 → RAG 检索 → LLM 生成答案
```

不包含：

* Tool Use；
* Permission；
* Verifier；
* Retry；
* Trace。

目的：

验证普通 RAG 在企业复杂任务上的不足。

### V1：RAG + Tool Use

流程：

```text
用户问题 → Planner / Router → RAG 或 Tool → LLM 生成答案
```

增加：

* query_sql；
* workflow_check；
* generate_report；
* Tool Contract 简化版。

目的：

验证工具调用对流程判断和数据分析任务的帮助。

### V2：RAG + Tool Use + Verifier

流程：

```text
V1 → Verifier 校验
```

增加：

* 引用检查；
* 工具调用检查；
* SQL 成功检查；
* 输出格式检查。

目的：

验证 Verifier 对缺引用、无依据回答和格式错误的拦截能力。

### V3：Full Harness Runtime

流程：

```text
V2 → Permission → Retry → Trace Logging
```

增加：

* 角色权限；
* 风险等级；
* retry / fallback；
* trace logging；
* error_type 分析。

目的：

验证完整 Harness Runtime 对任务完成率、可控性和可复盘性的提升。

## 7. 预估结果设计

注意：以下内容是**预估目标区间**，不是实际实验结果。
真实项目报告和简历中必须替换为实际跑出来的数据。

### 7.1 预估趋势

预期趋势：

```text
V0：知识问答可以完成，但流程判断、数据分析和权限控制能力弱。
V1：工具类任务明显提升，但仍可能缺引用或出现无依据结论。
V2：引用一致性和输出可靠性提升，但部分失败任务还不能自动修复。
V3：端到端任务成功率、权限控制和错误可复盘能力最好，但平均延迟会升高。
```

### 7.2 预估指标区间

| 指标                           | V0 Baseline RAG | V1 + Tool Use | V2 + Verifier | V3 Full Harness |
| ---------------------------- | --------------: | ------------: | ------------: | --------------: |
| RAG Recall@5                 |         70%-85% |       70%-85% |       70%-85% |         70%-85% |
| Tool Call Accuracy           |             不适用 |       70%-85% |       70%-85% |         75%-90% |
| Tool Success Rate            |             不适用 |       75%-90% |       75%-90% |         80%-95% |
| Citation Accuracy            |         55%-70% |       55%-75% |       70%-85% |         75%-90% |
| Task Success Rate            |         50%-65% |       60%-75% |       65%-80% |         70%-85% |
| Verifier Pass Rate           |             不适用 |           不适用 |       70%-85% |         75%-90% |
| Retry Success Rate           |             不适用 |           不适用 |           不适用 |         40%-70% |
| Permission Blocking Accuracy |             不适用 |           不适用 |           不适用 |        90%-100% |
| Average Latency              |              最低 |            中等 |            较高 |              最高 |

### 7.3 预估结果解释

#### RAG Recall@5

V0 到 V3 的 RAG Recall@5 预期变化不大。

原因：

RAG Recall@5 主要由文档切分、embedding、向量库和检索策略决定，Tool Use、Verifier、Permission 不会直接提升召回率。

如果后续加入 rerank 或 query rewrite，Recall@5 可能提升。

#### Tool Call Accuracy

V1 之后才有 Tool Call Accuracy。

预期 V3 可能高于 V1/V2，原因是：

* Tool Contract 更规范；
* Router 可以根据工具 schema 选择工具；
* Trace 分析可以帮助修复错误路由；
* Permission 可以拦截不合适的工具调用。

#### Citation Accuracy

V2 / V3 应明显高于 V0 / V1。

原因：

Verifier 会检查答案是否包含引用来源；缺失引用时触发 retry 或重新生成。

#### Task Success Rate

V3 应最高。

原因：

Full Harness 不仅有 RAG 和 Tool，还包含：

* Permission；
* Verifier；
* Retry；
* Fallback；
* Trace 分析。

这些机制可以减少任务中途失败和无依据输出。

#### Average Latency

V3 延迟会最高。

原因：

* 工具调用更多；
* Verifier 增加一次检查；
* Retry 可能导致重复检索或重新生成；
* Trace 写入本身开销较小，但整体执行步骤变多。

因此项目报告中应同时说明：

```text
Harness 提高任务成功率和可控性，但会带来一定延迟开销。
```

## 10. 评测脚本设计

## 10.1 eval_rag.py

输入：

```text
data/eval_tasks.jsonl
rag/retriever.py
```

输出：

```text
rag_recall_at_1
rag_recall_at_3
rag_recall_at_5
missed_cases
```

主要逻辑：

1. 读取包含 gold_docs 的任务；
2. 对每个 query 执行 retrieve；
3. 判断 top-k 是否包含 gold_docs；
4. 统计 Recall@K；
5. 输出未命中样本。

---

## 10.2 eval_tool.py

输入：

```text
logs/traces.jsonl
data/eval_tasks.jsonl
```

输出：

```text
tool_call_accuracy
tool_success_rate
wrong_tool_cases
permission_cases
```

主要逻辑：

1. 读取 trace；
2. 提取实际调用工具；
3. 与 expected_tools 对比；
4. 统计工具选择准确率；
5. 统计工具执行成功率；
6. 输出错误工具调用案例。

## 11. 预期错误类型与改进方向

| 错误类型              | 可能原因            | 改进方向                             |
| ----------------- | --------------- | -------------------------------- |
| retrieval_failure | query 与文档表述不一致  | query rewrite、rerank、优化 chunk    |
| wrong_tool        | Router 规则不足     | 增加规则、使用 LLM 分类、优化工具描述            |
| tool_error        | 参数缺失或 SQL 错误    | input_schema、参数校验、受控 SQL         |
| missing_citation  | 生成阶段忽略引用        | Context 约束、Verifier retry        |
| permission_denied | 用户权限不足          | 明确拒绝并提示权限要求                      |
| format_error      | 报告格式不稳定         | 模板化输出、format verifier            |
| ungrounded_answer | LLM 脱离证据生成      | 强化 evidence-only prompt、Verifier |
| latency_high      | 工具链过长或 retry 过多 | 限制最大步骤、缓存检索结果                    |

## 14. 评测方案结论

本项目评测不只看最终回答质量，而是围绕 Agent 执行链路进行分模块评估。

核心评测逻辑是：

```text
RAG 负责检索是否命中；
Tool Use 负责工具是否选对和执行成功；
Verifier 负责回答是否有依据；
Permission 负责是否拦截越权；
Trace 负责过程是否可复盘；
Task Success 负责端到端任务是否完成。
```

预期结果是：

1. Baseline RAG 在简单制度问答上表现尚可，但在流程判断、数据分析和权限控制任务上不足；
2. 加入 Tool Use 后，流程判断和数据分析任务成功率提升；
3. 加入 Verifier 后，引用一致性和无依据回答控制能力提升；
4. 加入 Retry、Permission 和 Trace Logging 后，任务完成率、权限拦截和错误复盘能力进一步提升；
5. Full Harness Runtime 会带来更高延迟，但换来更好的可控性、可观测性和端到端任务稳定性。

最终项目报告中应强调：

**Harness Engineering 的收益不是让底层模型参数变强，而是通过工具治理、执行校验、错误恢复和轨迹评估，让 Agent 在企业任务中更稳定、更可控、更容易复盘。**
