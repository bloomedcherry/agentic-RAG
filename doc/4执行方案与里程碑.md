## 1. 执行目标

本阶段目标是把项目从“设计文档”推进到“可运行 Demo”。

当前 Demo 不追求复杂企业级系统，而是优先跑通一个结构清晰、可展示、可评估的 Agentic RAG + Harness Runtime 闭环。

第一版需要实现：

```text
用户请求
→ Agent State
→ Planner / Router
→ Tool Contract / Registry
→ Permission
→ RAG 或 Tool 执行
→ Context Builder
→ LLM 生成答案
→ Verifier + Retry
→ Trace Logging
→ Eval 指标统计
```

最终产出包括：

1. 可运行的 Agent Demo；
2. 中等规模半真实企业知识库；
3. 模拟业务数据库；
4. 可插拔 Tool Contract；
5. Tool Registry；
6. RAG 检索子系统；
7. Planner / Router；
8. Permission / Safety 简化版；
9. Verifier / Retry 简化版；
10. Trace 日志；
11. Eval 测试集和评估脚本；
12. 项目复盘与简历素材。

---

## 2. 执行原则

### 2.1 先闭环，后增强

第一优先级不是功能多，而是跑通完整链路。

先实现：

```text
RAG + Tool + Runtime + Trace + Eval
```

再增强：

```text
Verifier + Retry + Permission + HITL
```

暂不实现：

```text
Skill
MCP
复杂 Multi-Agent
Agentic RL
真实企业系统接入
生产级监控
```

### 2.2 Demo 简洁，但架构不能写死

代码可以简化，但必须保留以下接口：

1. Tool Contract；
2. Tool Registry；
3. Agent State；
4. Trace Logger；
5. Retriever 接口；
6. Verifier 接口；
7. Eval 输入输出格式。

这样后续加 Multi-Agent、Skill、MCP 或轨迹偏好优化时，不需要大规模重构。

### 2.3 每个阶段都要有可验收产物

每个里程碑不能只写“完成开发”，而要有明确产物：

* 代码文件；
* 测试样例；
* 日志；
* 指标；
* Demo 截图；
* 文档。

# 5. M1：数据与 RAG 基础闭环

## 5.1 目标

构建中等规模半真实企业知识库，并实现最基础的 RAG 检索链路。

本阶段不做复杂 Agent，只保证：

```text
企业文档
→ 公开采样 / 模板扩展 / 人工精写
→ 解析
→ 切分
→ metadata 绑定
→ 向量化
→ 建索引
→ top-k 检索
→ 返回 source / chunk_id / score
```

M1 需要把 RAG 作为子系统实现，而不是只写一个 search_kb 函数。search_kb 是后续接入 Runtime 的工具接口；data_builder、parse_doc、chunker、build_index 和 retriever 是 RAG 子系统内部模块。

## 5.2 任务拆解

### 任务 1：构建中等规模半真实企业文档

建议准备 6 类文档：

```text
1. policies/：报销、采购、合同、人事、信息安全、数据权限
2. workflows/：采购审批、报销审批、合同审批、项目立项、风险升级
3. projects/：项目计划、周报、风险记录、需求变更、复盘
4. meetings/：会议纪要、待办、决策、阻塞点
5. contracts/：付款周期、交付、违约、保密、终止条款
6. reports/：周报模板、风险报告、经营分析、审批说明
```

文档规模建议：

```text
最低验收：300+ 文档，3,000+ chunks
较好验收：500+ 文档，5,000+ chunks
展示版本：800+ 文档，10,000+ chunks
```

数据来源建议：

```text
公开真实数据：30%-40%，例如公开 handbook / policy、公开合同数据、公开会议语料
模板扩展数据：50%-60%，按企业制度、流程、项目资料、会议纪要、报告模板批量生成
人工精写数据：10%，用于支撑 Demo 问题和 Eval Set
```

当前项目不接入真实企业内部系统，简历和报告中应写“公开真实数据 + 合成业务流程数据”，不要写“真实企业内部数据”。

示例：

```text
data/docs/
├── policies/
├── workflows/
├── projects/
├── meetings/
├── contracts/
└── reports/
```

### 任务 1.5：实现数据构建器

文件：

```text
rag/data_builder.py
```

功能：

* 从 data/raw/ 读取公开采样数据；
* 根据 data/generators/ 中的模板配置批量生成制度、流程、项目、会议、合同和报告文档；
* 写入标准化 Markdown 到 data/docs/；
* 输出 corpus_stats.json，记录文档数、类别分布和预估 chunk 数。

### 任务 2：实现文档解析

文件：

```text
tools/parse_doc.py
```

功能：

* 读取 Markdown / TXT；
* 提取标题；
* 提取正文；
* 生成 metadata。

第一版只支持 Markdown / TXT 即可。

### 任务 3：实现 chunk 切分

文件：

```text
rag/chunker.py
rag/build_index.py
```

每个 chunk 至少包含：

```json
{
  "chunk_id": "policy_reimbursement_001",
  "source": "policy_reimbursement.md",
  "doc_type": "policy",
  "title": "差旅报销制度",
  "content": "...",
```

### 任务 4：构建向量索引

可选方案：

如果只想快速完成 demo，用 Chroma 更省心。

文件：

```text

```

接口：

```python
def retrieve(query: str, top_k: int = 5, filters: dict | None = None) -> list[dict]:
    ...
```

返回：

```json
[
  {
    "chunk_id": "...",
```

# 6. M2：Tool Contract 与 Agent Runtime

## 6.1 目标

将 RAG、SQL、流程校验、报告生成封装为统一工具，并实现单 Supervisor Agent 的主执行流程。

本阶段要跑通：

```text
用户问题
→ Planner 判断任务类型
→ Router 选择工具
→ Tool 执行
→ Context Builder 组装结果
→ LLM 生成回答
```

## 6.2 任务拆解

### 任务 1：实现 BaseTool

文件：

```text
tools/base.py
```

建议字段：

```python
class BaseTool:
    name: str
    description: str
    input_schema: dict
    output_schema: dict
    permission: str
    risk_level: str
    timeout: int
    retry_policy: dict

    def run(self, **kwargs):
        raise NotImplementedError
```

文件：

```text
agent/planner.py
```

第一版使用规则分类：


输出：

```json
{
  "task_type": "workflow_query",
  "plan": ["search_kb", "workflow_check", "generate_answer"],

}
```

文件：

```text
agent/router.py
```

功能：

### 任务 9：实现 Context Builder

文件：

```text
agent/context_builder.py
```

功能：

* 拼接用户问题；

主流程：


# 7. M3：Verifier、Retry、Permission 与 Trace

## 7.1 目标

让系统从“能跑”升级为“可控、可观测、可复盘”。

本阶段加入：

```text
Permission / Safety
Verifier
Retry / Fallback
Trace Logging
```

## 7.2 任务拆解

### 任务 1：实现 Agent State

文件：

```text
agent/memory.py
```

状态结构：

```json
{
  "task_id": "...",
  "query": "...",
  "user_role": "employee",
  "task_type": "...",
  "plan": [],
  "tool_calls": [],
  "retrieved_docs": [],
  "tool_outputs": {},
  "answer": "",
  "verifier_result": {},
  "success": false,
  "errors": []
}
```

### 任务 2：实现 Permission

文件：

```text
agent/permission.py
```

第一版角色：

```text
employee
manager
admin
```

工具权限示例：

| 工具              | employee | manager | admin |
| --------------- | -------- | ------- | ----- |
| search_kb       | 是        | 是       | 是     |
| generate_report | 是        | 是       | 是     |
| workflow_check  | 是        | 是       | 是     |
| query_sql       | 否        | 是       | 是     |
| parse_doc       | 是        | 是       | 是     |

处理逻辑：

```text
权限通过 → 执行工具
权限不足 → permission_denied
高风险操作 → human-in-the-loop
```

### 任务 3：实现 Verifier

文件：

```text
agent/verifier.py
```

第一版用规则实现。

检查项：

1. RAG 任务是否有引用；
2. 工具调用是否成功；
3. SQL 失败时是否仍然生成了数据结论；
4. 权限不足时是否继续执行；
5. 报告格式是否包含必要标题；
6. 是否存在明显无依据结论。

输出格式：

```json
{
  "pass": false,
  "issues": [
    {
      "type": "missing_citation",
      "message": "回答缺少引用来源"
    }
  ],
  "suggested_action": "retry_with_citation"
}
```

### 任务 4：实现 Retry / Fallback

可以先写在：

```text
agent/runtime.py
```

后续再拆成：

```text
agent/retry.py
```

规则：

| 问题                | 动作               |
| ----------------- | ---------------- |
| missing_citation  | 重新生成，强制引用        |
| retrieval_empty   | 改写 query 重新检索    |
| sql_error         | fallback，不生成数据结论 |
| permission_denied | 拒绝执行并说明          |
| format_error      | 按模板重新生成          |
| unsupported_task  | 请求用户补充信息         |

### 任务 5：实现 Human-in-the-loop 简化版

第一版用命令行确认即可：

```text
该操作风险等级为 high，是否继续？y/n
```

如果做 Streamlit，可用按钮确认。

当前不做真实审批流。

### 任务 6：实现 Trace Logger

文件：

```text
agent/trace.py
```

写入：

```text
logs/traces.jsonl
```

每条 trace 包含：

```json
{
  "task_id": "uuid",
  "timestamp": "...",
  "query": "...",
  "user_role": "employee",
  "task_type": "knowledge_qa",
  "plan": [],
  "tool_calls": [],
  "retrieved_docs": [],
  "tool_outputs": {},
  "answer": "...",
  "verifier_result": {},
  "success": true,
  "latency": 2.31,
  "error_type": null
}
```

## 7.3 验收标准

M3 完成后：

1. 权限不足时，系统能拒绝 query_sql；
2. 回答缺少引用时，Verifier 能发现；
3. SQL 失败时，系统不应编造数据结论；
4. 每次请求都会生成 trace；
5. logs/traces.jsonl 可用于后续评估。


# 8. M4：Eval、Demo 展示与项目总结

## 8.1 目标

构建测试集和评估脚本，用量化指标证明 Harness 改造收益。

本阶段重点是：

```text
评估集
→ 执行 Agent
→ 生成 Trace
→ 统计指标
→ 分析失败类型
→ 输出项目总结
```

## 8.2 任务拆解

### 任务 1：构建 Eval Set

文件：

```text
data/eval_tasks.jsonl
```

第一版建议先做 50 条，展示版扩展到 80-100 条。

分布：

| 类型     | 数量 |
| ------ | -: |
| 企业制度问答 | 15 |
| 流程判断   | 10 |
| 项目资料分析 | 10 |
| 数据分析   | 10 |
| 报告生成   |  5 |

每条格式：

```json
{
  "query": "差旅报销需要哪些材料？",
  "user_role": "employee",
  "expected_task_type": "knowledge_qa",
  "expected_tools": ["search_kb"],
  "gold_docs": ["policy_reimbursement.md"],
  "expected_answer_points": ["发票", "审批单", "行程证明"],
  "risk_level": "low"
}
```

### 任务 2：实现 eval_rag.py

指标：

```text
RAG Recall@K
```

计算：

```text
Recall@K = top-k 中命中 gold_docs 的样本数 / 有 gold_docs 标注的样本数
```

### 任务 3：实现 eval_tool.py

指标：

```text
Tool Call Accuracy
Tool Success Rate
Permission Denied Accuracy
```

# 10. 风险与应对方案

## 10.1 RAG 检索效果差

表现：

* 检索不到正确文档；
* 检索结果不相关。

应对：

* 调整 chunk；
* 加 metadata；
* 调整 top_k；
* 增加 query rewrite；
* 后续加入 rerank。

## 10.2 工具调用错误

表现：

* Router 选错工具；
* 参数缺失；
* SQL 报错。

应对：

* Tool Contract；
* input_schema 校验；
* 规则 Router；
* 受控 SQL 模板；
* Retry / fallback。

## 10.3 输出没有引用

表现：

* Agent 根据常识回答，不引用文档。

应对：

* Context Builder 强约束；
* Verifier 检查；
* missing_citation 触发 retry。

## 10.4 权限边界不清楚

表现：

* 普通用户调用了 SQL 或高风险工具。

应对：

* role；
* permission；
* risk_level；
* permission_denied；
* HITL。

## 10.5 项目功能膨胀

表现：

* 想同时做 Multi-Agent、Skill、MCP、Agentic RL。

应对：

* 第一版只做单 Supervisor Agent；
* Skill / MCP / Multi-Agent / Agentic RL 只写扩展方向；
* 保留接口，不实现复杂功能。

# 14. 执行方案结论

3. 能展示 Tool Use；
4. 能展示权限与校验；
5. 能展示 Trace 和 Eval；
6. 能和微调项目形成区分；
