# 项目母版事实表

## 1. 项目名称

备选名称：

1. 基于 Harness Engineering 的企业 Agentic RAG 知识助手
2. 基于 Agentic RAG 的企业知识库问答与业务流程辅助系统
3. 企业知识库 Agentic RAG 与可控执行 Harness 系统

推荐使用：

**基于 Harness Engineering 的企业 Agentic RAG 知识助手**

原因：该名称能突出项目不是普通 RAG 问答，而是包含 Harness Runtime、Tool Use、Trace Logging、Verifier 和 Eval 的可控 Agent 系统。

---

## 2. 项目定位

该项目是一个面向企业内部知识管理与业务流程辅助场景的 Agentic RAG 应用项目。

当前版本定位为：

**单 Supervisor Agent + 可插拔 Tool/RAG + Harness Runtime**

项目重点不是训练大模型，而是构建一个可控、可观测、可扩展的 Agent 应用系统。

核心能力包括：

* RAG 知识库检索
* Tool Use 工具调用
* Agent Planning / Router
* Harness Runtime
* Memory
* Permission / Safety
* Verifier / Retry
* Trace Logging
* Eval 评估体系

---

## 3. 与微调项目的区别

该项目是第二个独立项目，不与第一个大模型微调项目混在一起。

| 对比项    | 微调项目                                    | 当前 Agent 项目                                      |
| ------ | --------------------------------------- | ------------------------------------------------ |
| 项目层次   | 模型层                                     | 应用层                                              |
| 核心目标   | 模型领域适配、偏好优化、评估和推理部署                     | 企业知识问答、流程辅助、工具调用和执行管控                            |
| 主要技术   | SFT、LoRA、DPO/PPO、LLM-as-Judge、vLLM、GPTQ | RAG、Tool Use、Planning、Harness Runtime、Trace、Eval |
| 是否训练模型 | 是                                       | 当前版本不训练模型                                        |
| 主要产出   | 微调模型、偏好数据、评估报告、推理压测结果                   | Agent 系统、工具调用链、执行轨迹、评估体系                         |
| 适合体现能力 | 模型训练与对齐能力                               | AI 应用落地与 Agent 工程能力                              |

一句话区分：

**微调项目解决“模型本身如何适配场景”，Agent 项目解决“模型如何结合知识、工具和流程完成企业任务”。**

## 5. 需求场景

项目场景定位为：

**企业内部知识管理与业务流程辅助**

不写成电商客服、售后退款、客服回复，避免和微调项目重复。

可使用场景包括：

### 5.1 企业制度问答

示例：

* 报销流程怎么走？
* 采购申请需要哪些材料？
* 合同审批需要经过哪些部门？

目标：根据制度文档进行知识问答，并给出引用依据。

### 5.2 流程判断

示例：

* 这个采购金额是否需要高级审批？
* 这个付款流程是否存在风险？
* 当前合同是否需要法务复核？

目标：结合制度文档和结构化数据判断流程路径。

### 5.3 项目资料分析

示例：

* 根据会议纪要总结项目风险。
* 根据项目文档整理待办事项。
* 汇总某项目近期进展和阻塞点。

目标：从项目资料中提取信息，辅助项目管理。

### 5.4 合同 / 制度条款检索

示例：

* 查找合同中关于付款周期的条款。
* 判断某项费用是否符合制度要求。

目标：实现条款级检索和引用溯源。

### 5.5 报告草稿生成

示例：

* 根据业务数据和制度要求生成审批材料。
* 根据项目资料生成项目周报。
* 根据检索结果和 SQL 查询结果生成分析报告。

目标：将检索证据和工具输出整合为结构化报告。

## 6. 目标用户

目标用户包括：

1. 企业普通员工

   * 查询制度、流程、报销、采购等知识。

2. 项目经理

   * 汇总项目资料、会议纪要、风险和待办事项。

3. 运营 / 数据分析人员

   * 查询业务数据并生成分析报告。

4. 管理者

   * 快速了解项目状态、流程风险和业务摘要。

5. 审批 / 流程管理人员

   * 判断申请是否符合制度要求。

---

## 7. 核心业务流程

系统核心流程：

1. 用户输入企业知识或业务流程相关问题。
2. Planner 判断任务类型。
3. Router 选择需要调用的能力。
4. RAG 或 Tool 执行。
5. Context Builder 组装用户问题、检索证据、工具结果和历史上下文。
6. LLM 生成答案或报告草稿。
7. Verifier 检查输出是否合规、有依据、无越权。
8. 如果校验失败，触发 retry、fallback、refusal 或 human-in-the-loop。
9. 输出最终结果。
10. Trace Logger 记录完整执行轨迹，用于后续评估和复盘。

---

## 8. 当前 Agent 架构

当前项目不是复杂 Multi-Agent 系统。

当前架构是：

```text
Single Supervisor Agent
  ├── RAG Tool
  ├── SQL Tool
  ├── Report Tool
  ├── Workflow Tool
  └── Verifier
```

当前系统可以理解为：

**单主控 Agent 负责规划和工具路由，Harness Runtime 负责执行状态、上下文、权限、校验和日志，底层通过 RAG 与业务工具完成任务。**

不应写成：

```text
Supervisor Agent
  ├── Research Agent
  ├── SQL Agent
  ├── Workflow Agent
  ├── Report Agent
  └── Verifier Agent
```

除非后续真的实现多 Subagent 调度。

## 9. Harness Runtime 模块

Harness Runtime 是项目核心。

它负责把 LLM 从“单次问答模型”变成“可执行、可管控、可追踪的 Agent 系统”。

包含模块：

1. Planner / Router

   * 判断任务类型；
   * 拆解简单执行步骤；
   * 决定是否需要 RAG 或 Tool。

2. Context Builder

   * 组装用户问题；
   * 加入历史上下文；
   * 加入检索结果；
   * 加入工具返回结果；
   * 控制 prompt 输入结构。

3. Tool Router

   * 根据任务选择工具；
   * 检查工具输入参数；
   * 管理工具调用结果。

4. Memory

   * 保存短期任务状态；
   * 长期记忆作为后续扩展。

5. Permission / Safety

   * 校验用户权限；
   * 拦截敏感操作；
   * 控制高风险工具调用。

6. Verifier

   * 检查答案是否有依据；
   * 检查是否越权；
   * 检查是否需要 retry。

7. Trace Logger

   * 记录任务执行全过程；
   * 为评估和复盘提供数据。

## 10. RAG 模块设计

RAG 模块用于企业知识增强。

主要对象：

* 企业制度文档
* 流程文档
* 项目资料
* 会议纪要
* 合同条款
* 历史报告

基本流程：

```text
文档解析
→ 文本切分
→ Embedding
→ 向量索引
→ Top-K 检索
→ 可选 Rerank
→ 返回证据片段
→ 引用溯源
```

当前版本必须实现：

* 文档加载
* chunk 切分
* 向量化
* top-k 检索
* 返回 source / chunk_id / score
* 支持答案引用

可简化实现：

* Rerank 可选；
* 混合检索可后续扩展；
* 大规模企业知识库暂不做。

## 11. Tool Use 模块设计

Tool Use 用于让 Agent 调用外部能力，而不是只生成文本。

当前计划工具：

1. search_kb

   * 知识库检索工具。

2. query_sql

   * 查询模拟业务数据库。

3. parse_doc

   * 文档解析工具。

4. generate_report

   * 报告生成工具。

5. workflow_check

   * 流程查询 / 流程校验工具。

工具调用目标：

* 让 Agent 能基于知识和数据回答；
* 降低无依据生成；
* 支持流程判断和报告生成；
* 为 Trace 和 Eval 提供可观测步骤。

---

## 12. Tool Contract 设计

Tool Contract 是简历加分模块之一。

每个工具需要统一定义：

* name
* description
* input_schema
* output_schema
* permission
* risk_level
* timeout
* retry_policy

目标：

1. 工具调用不是简单函数调用，而是可治理工具体系。
2. Router 可以根据 schema 选择工具。
3. Permission 模块可以根据 permission 和 risk_level 做限制。
4. Trace Logger 可以统一记录工具输入输出。
5. Verifier 可以检查工具是否被正确调用。
6. 后续可扩展新工具。

示例结构：

```python
class BaseTool:
    name: str
    description: str
    input_schema: dict
    output_schema: dict
    permission: str
    risk_level: str
    timeout: int
    retry_policy: dict

    def run(self, **kwargs):
        raise NotImplementedError
```

当前不引入 Skill / MCP。

但 Tool Contract 应该保留可扩展性，方便以后接入更多工具。

## 13. Memory 设计

当前 Memory 简化做。

优先实现短期任务状态：

* 当前用户问题
* 当前任务类型
* 当前 plan
* 已调用工具
* 工具返回结果
* 检索文档
* verifier 结果
* final answer
* error 信息

长期记忆作为后续扩展方向。

长期记忆可能包括：

* 用户偏好
* 常用报告格式
* 历史任务记录
* 用户所属角色
* 常用项目或部门

当前版本不强求长期记忆落地。

## 14. Permission / Safety 设计

当前版本做简化权限和安全机制。

重点包括：

1. 工具权限控制

   * 不同工具设置 permission。
   * 用户角色决定能否调用工具。

2. 风险等级控制

   * low：普通检索、普通问答。
   * medium：SQL 查询、报告生成。
   * high：流程提交、敏感数据导出。

3. 敏感操作拦截

   * 高风险操作需要 human-in-the-loop。
   * 权限不足时拒绝执行。

4. 输出安全

   * 避免无依据结论。
   * 避免越权访问信息。
   * 避免泄露敏感字段。

当前版本可以基于模拟角色实现：

* employee
* manager
* admin

具体权限矩阵【待确认】。

## 15. Verifier / Retry 设计

Verifier 是建议重点实现模块。

检查内容：

1. 回答是否有引用来源。
2. 是否调用了必要工具。
3. SQL 查询是否成功。
4. 是否存在无依据结论。
5. 是否存在越权访问。
6. 输出格式是否符合要求。

Verifier 输出：

* pass / fail
* issues
* suggested_action

失败后的处理方式：

1. retry

   * 重新检索；
   * 重写 query；
   * 重新生成答案；
   * 重新调用工具。

2. fallback

   * 降级为只基于知识库回答；
   * 提示用户补充信息。

3. human-in-the-loop

   * 高风险操作人工确认；
   * 不确定流程人工确认。

4. refusal

   * 权限不足或信息缺失时拒绝执行。

## 16. Trace Logging 设计

Trace Logging 是必须实现模块。

每次任务需要记录：

* query
* plan
* tool_calls
* retrieved_docs
* tool_outputs
* answer
* verifier_result
* success
* latency
* error_type

日志文件：

```text
logs/traces.jsonl
```

Trace 用途：

1. 复盘 Agent 决策过程。
2. 分析错误类型。
3. 计算评估指标。
4. 支撑简历中的量化实验。
5. 为后续 Multi-Agent / Agentic RL / DPO 轨迹优化提供数据基础。

当前不做 Agentic RL，但 Trace 可以作为后续扩展基础。

## 17. Eval 设计

Eval 是必须实现模块。

建议构造 30-100 条测试任务。

评估维度包括：

1. RAG Recall@K

   * 正确文档是否出现在 top-k 检索结果中。

2. Citation Accuracy

   * 最终回答引用是否能支撑结论。

3. Tool Call Accuracy

   * 是否选择了正确工具。

4. Tool Success Rate

   * 工具调用是否成功执行。

5. Task Success Rate

   * 端到端任务是否完成。

6. Verifier Pass Rate

   * 输出是否通过校验。

7. Average Latency

   * 平均响应时间。

8. Error Type Distribution

   * 错误类型分布，例如 retrieval_failure、tool_error、permission_denied、unsupported_task。

如果没有真实实验结果，只能写：

* 计划评估；
* 预期目标；
* 已构建评估脚本；
* 已构建评估集。

不能编造已经达到的指标。

## 19. 项目边界：必须做、简化做、暂不做

### 19.1 必须做

* RAG
* Tool Use
* Planner / Router
* Trace Logging
* Eval
* Tool Contract

### 19.2 建议做

* Verifier
* Retry
* Context Builder
* Permission / Safety 简化版
* Human-in-the-loop 简化版

* Memory

  * 先做短期任务状态。

  * 先做角色与工具权限映射。

  * 先做高风险操作确认。

## 20. 简历可写亮点

当前项目可写亮点：

1. 构建面向企业知识管理与业务流程辅助的 Agentic RAG 系统，支持制度问答、流程查询、项目资料分析和报告草稿生成。

2. 设计单 Supervisor Agent + Harness Runtime 架构，将任务规划、工具路由、上下文组装、权限校验、结果验证和 Trace Logging 统一封装。

3. 构建可插拔 Tool Contract，对知识库检索、SQL 查询、文档解析、流程校验和报告生成工具进行 schema 化封装，支持参数校验、权限控制、风险等级和执行审计。

4. 实现企业知识库 RAG 链路，完成文档解析、文本切分、Embedding、向量索引、Top-K 检索和引用溯源，提升回答的事实依据和可追踪性。

5. 设计 Verifier + Retry 机制，对答案引用、工具调用、SQL 执行、权限访问和输出格式进行校验，并在失败时触发重试、降级或人工确认。

6. 构建 Agent Trace Logging 与评估体系，记录 query、plan、tool_calls、retrieved_docs、answer、verifier_result 和 latency，用于统计 RAG Recall@K、Tool Call Accuracy、Task Success Rate 等指标。

## 21. 后续扩展方向

后续扩展方向包括：

1. Multi-Agent / Subagent

   * 将当前单 Supervisor Agent 扩展为 Supervisor + 专家 Agent 架构。
   * 专家 Agent 可包括 RAG Agent、SQL Agent、Workflow Agent、Report Agent、Verifier Agent。

2. Agentic RL / DPO 轨迹优化

   * 基于 Trace 构造成功 / 失败轨迹。
   * 将成功轨迹作为 chosen，失败轨迹作为 rejected。
   * 后续用于 DPO 或 Agentic RL。

3. Skill

   * 将高频任务封装为可复用 Skill。
   * 当前版本暂不做。

4. MCP

   * 通过 MCP 接入外部工具和资源。
   * 当前版本暂不做。

5. 更完整的权限系统

   * 扩展 RBAC；
   * 增加敏感字段脱敏；
   * 增加审计日志。

6. 更完整的评估体系

   * 扩展测试集规模；
   * 引入人工标注；
   * 引入 LLM-as-Judge；
   * 增加模块消融实验。

7. 真实业务系统接入

   * 接入真实数据库；
   * 接入企业文档系统；
   * 接入流程审批系统。

当前用户问题
当前任务类型

14. Permission / Safety 设计

当前版本做简化权限和安全机制。

重点包括：

工具权限控制
不同工具设置 permission。
用户角色决定能否调用工具。
风险等级控制
low：普通检索、普通问答。
medium：SQL 查询、报告生成。
high：流程提交、敏感数据导出。
敏感操作拦截
高风险操作需要 human-in-the-loop。
权限不足时拒绝执行。
输出安全
避免无依据结论。
避免越权访问信息。
避免泄露敏感字段。

当前版本可以基于模拟角色实现：

employee
manager
admin

具体权限矩阵【待确认】。

15. Verifier / Retry 设计

Verifier 是建议重点实现模块。

检查内容：

回答是否有引用来源。
是否调用了必要工具。
SQL 查询是否成功。
是否存在无依据结论。
是否存在越权访问。
输出格式是否符合要求。

Verifier 输出：

pass / fail
issues
suggested_action

失败后的处理方式：

retry
重新检索；
重写 query；
重新生成答案；
重新调用工具。
fallback
降级为只基于知识库回答；
提示用户补充信息。
human-in-the-loop
高风险操作人工确认；
不确定流程人工确认。
refusal
权限不足或信息缺失时拒绝执行。
16. Trace Logging 设计

Trace Logging 是必须实现模块。

每次任务需要记录：

query
plan
tool_calls
retrieved_docs
tool_outputs
answer
verifier_result
success
latency
error_type

日志文件：

logs/traces.jsonl

Trace 用途：

复盘 Agent 决策过程。
分析错误类型。
计算评估指标。
支撑简历中的量化实验。
为后续 Multi-Agent / Agentic RL / DPO 轨迹优化提供数据基础。

当前不做 Agentic RL，但 Trace 可以作为后续扩展基础。

17. Eval 设计

Eval 是必须实现模块。

建议构造 30-100 条测试任务。

评估维度包括：

RAG Recall@K
正确文档是否出现在 top-k 检索结果中。
Citation Accuracy
最终回答引用是否能支撑结论。
Tool Call Accuracy
是否选择了正确工具。
Tool Success Rate
工具调用是否成功执行。
Task Success Rate
端到端任务是否完成。
Verifier Pass Rate
输出是否通过校验。
Average Latency
平均响应时间。
Error Type Distribution
错误类型分布，例如 retrieval_failure、tool_error、permission_denied、unsupported_task。

如果没有真实实验结果，只能写：

计划评估；
预期目标；
已构建评估脚本；
已构建评估集。

不能编造已经达到的指标。

│   ├── docs/                     # 企业制度、流程、项目文档

20. 简历可写亮点

当前项目可写亮点：

构建面向企业知识管理与业务流程辅助的 Agentic RAG 系统，支持制度问答、流程查询、项目资料分析和报告草稿生成。
设计单 Supervisor Agent + Harness Runtime 架构，将任务规划、工具路由、上下文组装、权限校验、结果验证和 Trace Logging 统一封装。
构建可插拔 Tool Contract，对知识库检索、SQL 查询、文档解析、流程校验和报告生成工具进行 schema 化封装，支持参数校验、权限控制、风险等级和执行审计。
实现企业知识库 RAG 链路，完成文档解析、文本切分、Embedding、向量索引、Top-K 检索和引用溯源，提升回答的事实依据和可追踪性。
设计 Verifier + Retry 机制，对答案引用、工具调用、SQL 执行、权限访问和输出格式进行校验，并在失败时触发重试、降级或人工确认。
构建 Agent Trace Logging 与评估体系，记录 query、plan、tool_calls、retrieved_docs、answer、verifier_result 和 latency，用于统计 RAG Recall@K、Tool Call Accuracy、Task Success Rate 等指标。
