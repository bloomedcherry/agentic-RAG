## 1. 项目名称

推荐名称：

**基于 Harness Engineering 的企业 Agentic RAG 知识助手**

备选名称：

**基于 Agentic RAG 的企业知识库问答与业务流程辅助系统**

## 2. 项目定位

本项目是一个面向企业内部知识管理与业务流程辅助场景的 Agentic RAG 应用项目。

当前版本不是复杂 Multi-Agent 系统，而是：

**单 Supervisor Agent + 可插拔 Tool/RAG + Harness Runtime**

项目重点不在于训练大模型，而在于构建一个能够结合企业知识库、业务工具、流程约束和执行轨迹的可控 Agent 系统。

核心目标是让大模型不再只是单轮问答，而是能够：

* 检索企业制度、项目文档、会议纪要等内部知识；
* 调用 SQL、流程校验、报告生成等业务工具；
* 根据任务类型进行简单规划和工具路由；
* 对回答进行引用校验、权限校验和必要的重试；
* 记录完整执行轨迹，用于评估、复盘和后续扩展。

## 3. 与第一个微调项目的区别

本项目是第二个独立项目，不与第一个“大模型微调项目”混在一起。

第一个项目偏模型层，重点是：

* Qwen3-8B 微调；
* SFT；
* 偏好数据构造；
* 模型评测；
* vLLM / GPTQ 推理优化。

本项目偏应用层，重点是：

* RAG；
* Tool Use；
* Agent Planning；
* Harness Runtime；
* Permission / Safety；
* Verifier；
* Trace Logging；
* Eval。

两者关系可以概括为：

**微调项目解决模型本身如何适配任务，Agent 项目解决模型如何结合知识、工具和流程完成企业任务。**

## 4. 项目背景

在企业内部场景中，大量信息分散在制度文档、流程说明、项目资料、会议纪要、合同条款和业务数据库中。员工或管理者在处理日常工作时，经常需要跨多个信息源查找资料、判断流程、分析数据并生成材料。

传统方式通常存在以下问题：

1. 信息分散
   制度、流程、项目记录和业务数据分布在不同文件或系统中，用户需要手动查找。

2. 查询成本高
   用户不一定知道关键词、文档位置或具体流程名称，导致检索效率低。

3. 普通 RAG 只能回答，不能执行
   普通 RAG 可以检索文档并回答问题，但无法进一步调用 SQL、流程校验、报告生成等工具。

4. 大模型容易无依据生成
   如果没有引用校验和工具调用约束，模型可能生成没有依据的判断或建议。

5. 企业场景需要权限控制
   不同角色能访问的知识和工具不同，高风险操作需要人工确认。

6. Agent 执行过程难以复盘
   如果不记录 plan、tool_calls、retrieved_docs 和 verifier_result，很难评估 Agent 到底哪里出了问题。

因此，本项目希望构建一个轻量但完整的企业 Agentic RAG Demo，通过 Harness Runtime 把 RAG、工具调用、权限校验、结果验证和 Trace Logging 组织起来，形成一个可控、可评估、可扩展的 Agent 应用系统。

## 5. 项目目标

### 5.1 业务目标

本项目的业务目标是：

1. 提升企业内部知识查询效率；
2. 辅助员工理解制度和流程；
3. 支持项目资料分析和风险总结；
4. 支持基于知识库与结构化数据生成报告草稿；
5. 降低大模型在企业知识问答中的无依据生成风险；
6. 为后续 Multi-Agent、Agentic RL、Skill、MCP 等扩展预留工程基础。

### 5.2 技术目标

本项目的技术目标是：

1. 构建企业知识库 RAG 检索链路；
2. 设计统一 Tool Contract，使工具调用可治理、可扩展、可审计；
3. 实现简单 Planner / Router，根据任务类型选择 RAG 或工具；
4. 实现 Context Builder，将用户问题、检索结果、工具结果和任务状态组织进上下文；
5. 实现 Permission / Safety，对工具权限和高风险操作进行简化控制；
6. 实现 Verifier / Retry，对回答引用、工具调用和权限访问进行校验；
7. 实现 Trace Logging，记录 Agent 执行过程；
8. 构建 Eval Set，评估 RAG、Tool Use、Task Success 和 Latency 等指标。

## 9. 为什么普通 RAG 不够

普通 RAG 的流程通常是：

```text
用户问题 → 检索知识库 → 拼接上下文 → 生成回答
```

它可以解决“知识库问答”问题，但对企业流程辅助场景还不够。

不足包括：

1. 不能主动判断任务类型
   例如数据分析任务需要 SQL 工具，而不是只检索文档。

2. 不能稳定调用业务工具
   例如流程判断、数据查询、报告生成需要工具链。

3. 缺少权限控制
   不同用户角色不能访问相同数据或工具。

4. 缺少执行轨迹
   普通 RAG 不一定记录 plan、tool_calls 和 verifier_result。

5. 缺少失败恢复
   检索为空、SQL 报错、答案无引用时，需要 retry 或 fallback。

因此，本项目采用 Agentic RAG + Harness Runtime 的设计，使系统不仅能“检索后回答”，还可以“规划、调用工具、校验结果、记录轨迹”。

## 10. 为什么需要 Harness Runtime

Harness Runtime 的作用是把大模型的推理能力包在一个可控执行框架中。

它解决的问题包括：

1. 任务如何拆解；
2. 工具如何选择；
3. 上下文如何组装；
4. 权限如何校验；
5. 输出如何验证；
6. 失败如何恢复；
7. 过程如何记录；
8. 效果如何评估。

如果没有 Harness Runtime，Agent 很容易退化成：

```text
一个 prompt + 若干工具函数 + 不可控的模型输出
```

加入 Harness Runtime 后，系统变成：

```text
任务规划 → 工具调用 → 上下文组装 → 结果校验 → 失败恢复 → Trace 记录 → Eval 分析
```

这也是本项目区别于普通 RAG Demo 的关键。

## 11. 功能性需求

### 11.1 RAG 知识库检索

系统应支持：

* 加载企业制度、流程文档、项目资料、会议纪要等文档；
* 对文档进行 chunk 切分；
* 使用 embedding 构建向量索引；
* 根据用户 query 进行 top-k 检索；
* 返回文档片段、source、chunk_id 和 score；
* 支持最终回答引用来源。

优先级：必须。

### 11.2 Planner / Router

系统应支持：

* 判断用户任务类型；
* 任务类型包括知识问答、流程查询、数据分析、报告生成；
* 根据任务类型选择 RAG 或工具；
* 生成简单 plan；
* 将 plan 写入 Trace。

优先级：必须。

### 11.3 Tool Use

系统应支持以下工具：

* search_kb：知识库检索；
* query_sql：查询模拟业务数据库；
* parse_doc：解析文档；
* generate_report：生成报告草稿；
* workflow_check：流程校验。

优先级：必须。

### 11.4 Tool Contract

每个工具应有统一定义：

* name；
* description；
* input_schema；
* output_schema；
* permission；
* risk_level；
* timeout；
* retry_policy。

优先级：必须。

### 11.5 Context Builder

系统应支持将以下内容组织进模型上下文：

* 用户问题；
* 历史上下文；
* 当前 plan；
* 检索证据；
* 工具返回结果；
* 输出格式要求；
* 安全约束。

优先级：建议。

### 11.6 Memory

当前版本优先实现短期任务状态，包括：

* query；
* plan；
* tool_calls；
* retrieved_docs；
* tool_outputs；
* verifier_result；
* final_answer；
* errors。

长期记忆作为后续扩展。

优先级：简化做。

### 11.7 Permission / Safety

系统应支持：

* 用户角色；
* 工具级权限；
* 风险等级；
* 权限不足拒绝；
* 高风险操作人工确认；
* 敏感操作拦截。

优先级：简化做。

### 11.8 Verifier / Retry

系统应支持检查：

* 回答是否有引用来源；
* 是否调用了必要工具；
* SQL 是否执行成功；
* 是否存在无依据结论；
* 是否越权；
* 输出格式是否符合要求。

如果不通过，应支持：

* retry；
* fallback；
* human-in-the-loop；
* refusal。

优先级：建议。

### 11.9 Trace Logging

系统应记录：

* query；
* plan；
* tool_calls；
* retrieved_docs；
* tool_outputs；
* answer；
* verifier_result；
* success；
* latency；
* error_type。

优先级：必须。

### 11.10 Eval Set

系统应构建 50-100 条测试任务，评估：

* RAG Recall@K；
* Citation Accuracy；
* Tool Call Accuracy；
* Tool Success Rate；
* Task Success Rate；
* Verifier Pass Rate；
* Average Latency；
* Error Type Distribution。

优先级：必须。

* 引用来源；
* Verifier；
* Permission；
* Human-in-the-loop；
* refusal。

### 12.2 可观测性

体现方式：

* traces.jsonl；
* plan；
* tool_calls；
* retrieved_docs；
* verifier_result；
* error_type；
* latency。

### 12.3 可扩展性

系统应支持后续扩展新工具和新能力。

体现方式：

* Tool Contract；
* Tool Registry；
* 统一 input_schema / output_schema；
* 可插拔 RAG / SQL / Workflow / Report Tool。

当前版本不实现 Skill / MCP，但 Tool Contract 设计应保留后续扩展空间。

### 12.4 可评估性

系统应通过 eval 脚本量化模块效果。

体现方式：

* eval_rag.py；
* eval_tool.py；
* eval_task.py；
* analyze_trace.py；
* 评估集；
* 指标统计。

### 12.5 安全性

系统应对敏感工具和高风险操作做限制。

体现方式：

* permission；
* risk_level；
* role；
* HITL；
* permission_denied 错误类型。

### 12.6 Demo 简洁性

系统当前是 demo 项目，不做复杂企业级平台。

不做：

* 完整 Multi-Agent；
* Agentic RL；
* Skill；
* MCP；
* 真实生产级监控；
* 真实企业系统接入；
* 大规模权限系统。

## 15. 主要风险与约束

### 15.1 数据真实性风险

当前使用公开真实数据、模板扩展数据、人工精写样例和模拟业务数据库，不能直接宣称接入真实企业内部系统。

处理方式：

* 简历中写“公开真实数据 + 合成业务流程数据 + 模拟业务数据库”；
* 不写“上线生产系统”。

### 15.2 评估结果不足风险

如果尚未跑完评估，不能写具体提升百分比。

处理方式：

* 写“构建评估体系”；
* 写“计划统计指标”；
* 实验完成后再补充实际数值。

### 15.3 项目边界膨胀风险

如果同时写 Multi-Agent、Skill、MCP、Agentic RL，项目会显得过度包装。

处理方式：

* 当前版本只写单 Supervisor Agent；
* 其他能力只写后续扩展。

### 15.4 工具调用可靠性风险

模型可能选择错误工具或传入错误参数。

处理方式：

* Tool Contract；
* input_schema；
* verifier；
* retry；
* trace 分析。

### 15.5 无依据生成风险

模型可能没有检索到证据却直接回答。

处理方式：

* Citation Accuracy；
* Verifier；
* refusal；
* fallback。

### 15.6 权限风险

模型可能调用用户无权访问的工具或数据。

处理方式：

* role；
* permission；
* risk_level；
* HITL；
* permission_denied。

## 19. 需求分析结论

本项目的核心需求不是做一个普通知识库问答系统，而是构建一个带 Harness Runtime 的企业 Agentic RAG Demo。

项目要体现的重点是：

1. 通过 RAG 解决企业知识检索问题；
2. 通过 Tool Use 支持 SQL、流程校验和报告生成；
3. 通过 Planner / Router 实现简单任务分发；
4. 通过 Tool Contract 让工具调用可治理、可扩展；
5. 通过 Verifier / Retry 降低无依据回答和工具调用失败；
6. 通过 Permission / Safety 体现企业场景的安全边界；
7. 通过 Trace Logging 和 Eval 让 Agent 执行过程可评估、可复盘。

当前版本应控制复杂度，不做 Skill、MCP、完整 Multi-Agent 或 Agentic RL。项目后续可以基于 Trace 和 Tool Contract 继续扩展，但第一版目标应是跑通一个结构清晰、可展示、可量化的 Agentic RAG + Harness Runtime demo。
