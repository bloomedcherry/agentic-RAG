# 02_技术路线报告

## 1. 技术路线总览

本项目的技术路线可以概括为：

```text
企业文档与模拟业务数据
→ RAG 知识库构建
→ Tool Contract 工具体系
→ Planner / Router 任务分发
→ Harness Runtime 执行编排
→ Context Builder 上下文组装
→ LLM 生成答案或报告
→ Verifier 校验与 Retry
→ Trace Logging 执行记录
→ Eval 指标评估
```

一句话概括：

**系统以单 Supervisor Agent 为主控，通过 Harness Runtime 统一管理 RAG 检索、工具调用、上下文构建、权限校验、结果验证和执行轨迹记录，使大模型能够在企业知识问答、流程判断、数据分析和报告生成场景中完成可追踪、可评估的任务执行。**

当前项目不是复杂 Multi-Agent 系统，也不做 Agentic RL、Skill 或 MCP。
当前重点是先完成一个结构清晰、边界明确、可展示、可量化评估的 Agentic RAG Demo。

---

## 2. 总体技术架构

系统整体分为五层：

```text
入口层
→ Agent Harness Runtime 层
→ 能力层
→ 数据层
→ 评估与观测层
```

### 2.1 入口层

入口层负责接收用户请求，并将请求传入 Agent Runtime。

可选入口包括：

* CLI；
* FastAPI；
* Streamlit；
* FastAPI + Streamlit。

第一版建议：

```text
Streamlit 或 CLI 优先，用于快速展示 Demo；
FastAPI 可作为后续服务化接口。
```

入口层输入：

```text
用户问题
用户角色
可选上下文
可选文件
```

入口层输出：

```text
Agent 最终回答
引用来源
工具调用摘要
执行 Trace
```

### 2.2 Agent Harness Runtime 层

这是项目核心层。

主要模块包括：

* Planner / Router；
* Context Builder；
* Tool Router；
* Memory；
* Permission / Safety；
* Verifier；
* Retry / Fallback；
* Trace Logger。

Runtime 层负责把用户请求转化为一个可执行任务，并管理任务执行的完整生命周期。

### 2.3 能力层

能力层提供 Agent 可调用的具体能力。

当前版本包括：

* RAG 知识库检索；
* SQL / 数据查询工具；
* 文档解析工具；
* 流程查询 / 流程校验工具；
* 报告生成工具。

这些能力通过 Tool Contract 接入 Runtime，不直接暴露给用户。

### 2.4 数据层

数据层包括：

* 企业制度文档；
* 流程文档；
* 项目资料；
* 会议纪要；
* 合同条款；
* 模拟业务数据库；
* 向量索引；
* Trace 日志；
* Eval 测试集。

当前版本使用模拟企业数据，不接入真实企业系统。

### 2.5 评估与观测层

评估与观测层负责衡量 Agent 的效果和定位失败原因。

包括：

* traces.jsonl；
* eval_rag.py；
* eval_tool.py；
* eval_task.py；
* analyze_trace.py。

评估指标包括：

* RAG Recall@K；
* Citation Accuracy；
* Tool Call Accuracy；
* Tool Success Rate；
* Task Success Rate；
* Verifier Pass Rate；
* Average Latency；
* Error Type Distribution。

## 3. 系统执行流程

完整执行流程如下：

```text
1. 用户输入问题
2. Planner 判断任务类型
3. Router 选择 RAG / Tool / Report 等执行路径
4. Permission 检查用户是否有权限调用对应工具
5. RAG 或 Tool 执行
6. Context Builder 组装证据、工具结果和历史状态
7. LLM 生成答案或报告草稿
8. Verifier 检查答案是否有引用、是否越权、是否存在无依据结论
9. 如果校验失败，触发 retry / fallback / refusal / human-in-the-loop
10. 输出最终结果
11. Trace Logger 记录完整执行轨迹
12. Eval 脚本基于 Trace 和测试集计算指标
```

对应流程图：

```text
User Query
  ↓
Planner / Router
  ↓
Permission Check
  ↓
RAG / Tool Execution
  ↓
Context Builder
  ↓
LLM Generation
  ↓
Verifier
  ├── Pass → Final Answer
  └── Fail → Retry / Fallback / Refusal / HITL
  ↓
Trace Logging
  ↓
Evaluation
```

## 4. 任务类型与执行路径

Planner / Router 需要将用户请求分为不同任务类型，并选择对应执行路径。

| 任务类型   | 示例问题          | 执行路径                               | 主要工具                                  |
| ------ | ------------- | ---------------------------------- | ------------------------------------- |
| 知识问答   | 报销流程需要哪些材料？   | RAG 检索 → 生成回答 → 引用校验               | search_kb                             |
| 流程查询   | 采购金额超过多少需要审批？ | RAG 检索 → workflow_check → 回答       | search_kb, workflow_check             |
| 数据分析   | 本月销售为什么下降？    | SQL 查询 → RAG 补充背景 → 报告生成           | query_sql, search_kb, generate_report |
| 项目资料分析 | 总结 A 项目风险     | RAG 检索项目文档 → 汇总分析                  | search_kb, generate_report            |
| 报告生成   | 根据资料生成周报      | RAG / SQL → generate_report → 格式校验 | search_kb, query_sql, generate_report |
| 权限敏感任务 | 导出敏感数据        | Permission → HITL / Refusal        | permission, human-in-the-loop         |

第一版 Planner 可以先用规则或 LLM 分类实现，不需要复杂规划算法。

## 5. RAG 技术路线

### 5.1 RAG 目标

RAG 模块解决的问题是：

```text
让 Agent 回答企业知识问题时有依据，而不是凭模型记忆或常识生成。
```

应用场景包括：

* 企业制度问答；
* 流程条款检索；
* 项目资料分析；
* 会议纪要总结；
* 合同条款定位；
* 报告生成中的背景资料补充。

### 5.2 文档处理流程

```text
原始文档
→ 文档解析
→ 文本清洗
→ chunk 切分
→ metadata 绑定
→ embedding 向量化
→ 向量索引构建
```

文档类型：

* Markdown；
* TXT；
* PDF；
* Word；
* CSV / 表格类资料。

第一版建议优先支持：

```text
Markdown / TXT / CSV
```

PDF / Word 可后续扩展。

### 5.3 Chunk 设计

每个 chunk 至少包含：

```json
{
  "chunk_id": "policy_001_chunk_003",
  "source": "报销制度.md",
  "doc_type": "policy",
  "title": "差旅报销制度",
  "content": "...",
  "metadata": {
    "department": "finance",
    "created_at": "2026-xx-xx"
  }
}
```

切分策略：

* 按标题 / 段落切分；
* 控制 chunk 长度；
* 保留 source 和 chunk_id；
* 避免切断制度条款。

第一版可以采用固定长度 + 标题层级的简单策略。

### 5.4 检索流程

```text
用户 query
→ query embedding
→ top-k 向量检索
→ 返回候选 chunks
→ 可选 rerank
→ 交给 Context Builder
```

返回字段：

```json
{
  "chunk_id": "...",
  "source": "...",
  "content": "...",
  "score": 0.82
}
```

### 5.5 Rerank 设计

Rerank 当前为可选模块。

第一版可以先不做 rerank，只做向量 top-k。

如果后续加入 rerank，可以采用：

```text
向量召回 top-20
→ reranker 精排
→ 取 top-5 进入上下文
```

### 5.6 RAG 输出要求

RAG 不直接生成最终答案，只提供证据。

最终回答必须引用 RAG 返回的 source 或 chunk_id。

Verifier 需要检查：

* 回答是否引用来源；
* 引用是否能支撑答案；
* 是否存在无依据结论。

## 6. Tool Use 技术路线

### 6.1 Tool Use 目标

Tool Use 解决的问题是：

```text
让 Agent 不只会检索和生成，还能调用业务工具完成数据查询、流程校验和报告生成。
```

当前计划工具包括：

* search_kb；
* query_sql；
* parse_doc；
* generate_report；
* workflow_check。

### 6.2 工具调用链路

```text
Planner 判断任务
→ Router 选择工具
→ Permission 检查权限
→ Tool Contract 校验参数
→ Tool 执行
→ 返回结构化结果
→ Trace 记录
→ Context Builder 组装结果
```

### 6.3 Tool Contract

每个工具必须统一定义：

```python
class BaseTool:
    name: str
    description: str
    input_schema: dict
    output_schema: dict
    permission: str
    risk_level: str
    timeout: int
    retry_policy: dict

    def run(self, **kwargs):
        raise NotImplementedError
```

Tool Contract 的作用：

1. 支持工具注册；
2. 支持 Router 根据 description 选择工具；
3. 支持参数校验；
4. 支持权限控制；
5. 支持风险等级判断；
6. 支持执行日志记录；
7. 支持后续扩展新工具。

### 6.4 工具定义示例

#### search_kb

```text
功能：检索企业知识库
输入：query, top_k, filters
输出：retrieved_docs
权限：employee
风险等级：low
```

#### query_sql

```text
功能：查询模拟业务数据库
输入：query_type, filters 或 sql
输出：table_result, summary
权限：manager / analyst
风险等级：medium
```

第一版建议不要让模型自由生成任意 SQL，可以先采用预设查询模板或受控 SQL。

#### workflow_check

```text
功能：根据流程规则判断是否需要审批或人工确认
输入：process_type, amount, department, user_role
输出：decision, reason, required_approval
权限：employee / manager
风险等级：medium
```

#### generate_report

```text
功能：根据检索证据和工具结果生成报告草稿
输入：task_type, evidence, tool_outputs, format
输出：markdown_report
权限：employee
风险等级：low / medium
```

#### parse_doc

```text
功能：解析上传或指定文档
输入：file_path 或 doc_id
输出：text_chunks, metadata
权限：employee
风险等级：low
```

## 7. Planner / Router 技术路线

### 7.1 Planner 职责

Planner 负责判断用户问题属于哪类任务，并生成简单执行计划。

输入：

```text
query
user_role
session_state
```

输出：

```json
{
  "task_type": "knowledge_qa",
  "plan": [
    "search_kb",
    "generate_answer",
    "verify_citation"
  ],
  "required_tools": ["search_kb"]
}
```

### 7.2 Router 职责

Router 负责根据 Planner 输出选择具体工具。

例如：

```text
知识问答 → search_kb
流程判断 → search_kb + workflow_check
数据分析 → query_sql + generate_report
报告生成 → search_kb + query_sql + generate_report
```

### 7.3 第一版实现方式

第一版建议采用：

```text
规则匹配 + 少量 LLM 分类
```

例如：

* 包含“制度、流程、怎么走、需要哪些材料” → knowledge_qa；
* 包含“是否需要审批、是否符合制度” → workflow_check；
* 包含“数据、销售、下降、增长、统计” → data_analysis；
* 包含“生成报告、周报、总结” → report_generation。

后续可以替换为更复杂的 LLM planner。

## 8. Context Builder 技术路线

### 8.1 Context Builder 目标

Context Builder 负责把不同来源的信息组织成稳定的模型输入。

输入包括：

* 用户问题；
* 系统指令；
* 任务类型；
* plan；
* RAG 检索结果；
* 工具返回结果；
* Memory；
* 权限信息；
* 输出格式要求。

输出是 LLM prompt。

### 8.2 上下文结构

建议采用固定结构：

```text
[System Instruction]
你是企业知识助手，必须基于证据和工具结果回答。

[User Role]
当前用户角色：employee / manager / admin

[Task]
用户问题：...

[Plan]
当前任务计划：...

[Evidence]
检索证据：
1. source: xxx, chunk_id: xxx, content: xxx
2. source: xxx, chunk_id: xxx, content: xxx

[Tool Results]
工具返回：
query_sql: ...
workflow_check: ...

[Constraints]
- 不要编造未在证据中出现的信息
- 如果证据不足，说明无法确认
- 高风险建议需要提示人工确认
- 回答必须包含引用来源

[Output Format]
...
```

### 8.3 Context 控制策略

需要控制：

* 检索结果数量；
* 单个 chunk 长度；
* 工具输出长度；
* 历史上下文长度；
* 输出格式。

第一版可以简单实现：

```text
top_k = 3 或 5
每个 chunk 截断到固定长度
只保留最近一轮任务状态
```

## 9. Memory 技术路线

### 9.1 当前 Memory 定位

当前版本只做简化 Memory，重点是短期任务状态，不做复杂长期记忆。

短期 Memory 保存：

```json
{
  "query": "...",
  "task_type": "...",
  "plan": [],
  "tool_calls": [],
  "retrieved_docs": [],
  "tool_outputs": [],
  "verifier_result": {},
  "final_answer": "",
  "errors": []
}
```

### 9.2 Memory 用途

短期 Memory 用于：

* 支持多步任务执行；
* 给 Verifier 提供上下文；
* 给 Retry 提供失败原因；
* 写入 Trace；
* 支撑 Eval 分析。

长期 Memory 后续可扩展：

* 用户偏好；
* 历史任务；
* 常用报告格式；
* 用户所属部门；
* 常用项目。

当前不作为第一版重点。

## 10. Permission / Safety 技术路线

### 10.1 权限目标

Permission / Safety 解决企业场景中的安全边界问题。

需要控制：

* 谁能调用什么工具；
* 哪些操作需要人工确认；
* 哪些数据不能返回；
* 权限不足时如何拒绝。

### 10.2 用户角色

第一版可以定义三个角色：

```text
employee
manager
admin
```

### 10.3 工具风险等级

工具风险等级：

```text
low：普通知识库检索、普通问答
medium：SQL 查询、流程判断、报告生成
high：敏感数据导出、流程提交、外部系统写操作
```

当前版本主要做 low / medium。

high 风险操作可以作为 HITL 示例，不接真实系统。

### 10.4 权限检查流程

```text
Router 选择工具
→ 读取工具 permission 和 risk_level
→ 检查 user_role 是否允许调用
→ 如果不允许，返回 permission_denied
→ 如果高风险，触发 human-in-the-loop
→ 如果允许，执行工具
```

### 10.5 安全失败处理

失败类型包括：

* permission_denied；
* sensitive_data_blocked；
* high_risk_need_approval；
* unsupported_operation。

这些 error_type 需要写入 Trace。

## 11. Verifier / Retry 技术路线

### 11.1 Verifier 目标

Verifier 用于检查 Agent 输出是否满足企业场景要求。

检查项：

1. 是否有引用来源；
2. 引用是否能支撑答案；
3. 是否调用了必要工具；
4. SQL 查询是否成功；
5. 是否存在无依据结论；
6. 是否越权；
7. 输出格式是否正确。

### 11.2 Verifier 输出

```json
{
  "pass": false,
  "issues": [
    {
      "type": "missing_citation",
      "message": "回答中缺少引用来源"
    }
  ],
  "suggested_action": "retry_with_citation"
}
```

### 11.3 Retry 策略

根据不同失败类型触发不同动作。

| 失败类型              | 处理方式                    |
| ----------------- | ----------------------- |
| missing_citation  | 重新生成并强制引用               |
| retrieval_empty   | query rewrite 后重新检索     |
| sql_error         | 根据错误信息重试 SQL 或 fallback |
| permission_denied | 拒绝执行并说明原因               |
| unsupported_task  | 请求用户补充信息或拒绝             |
| format_error      | 按目标格式重新生成               |
| ungrounded_answer | 重新基于 evidence 生成        |

### 11.4 第一版实现方式

第一版 Verifier 可以采用：

```text
规则校验 + 简单 LLM-as-Judge 可选
```

优先做规则：

* 如果任务使用 RAG，答案必须包含 source；
* 如果 query_sql 失败，不能生成数据结论；
* 如果权限不足，不能继续调用工具；
* 如果报告格式不符合要求，触发重写。

## 12. Trace Logging 技术路线

### 12.1 Trace 目标

Trace Logging 用于记录 Agent 全流程，使系统可观测、可评估、可复盘。

每个任务记录一条 trace。

### 12.2 Trace 字段

```json
{
  "task_id": "uuid",
  "timestamp": "...",
  "query": "...",
  "user_role": "employee",
  "task_type": "knowledge_qa",
  "plan": [],
  "tool_calls": [
    {
      "tool_name": "search_kb",
      "input": {},
      "output_summary": "...",
      "status": "success",
      "latency": 0.42
    }
  ],
  "retrieved_docs": [
    {
      "source": "报销制度.md",
      "chunk_id": "policy_001_chunk_003",
      "score": 0.82
    }
  ],
  "tool_outputs": {},
  "answer": "...",
  "verifier_result": {
    "pass": true,
    "issues": []
  },
  "success": true,
  "latency": 3.21,
  "error_type": null
}
```

### 12.3 Trace 用途

Trace 用于：

1. 分析 Agent 执行过程；
2. 统计工具调用成功率；
3. 计算任务完成率；
4. 统计错误类型；
5. 分析 Verifier 是否有效；
6. 支撑简历中的量化实验；
7. 后续构造成功 / 失败轨迹数据。

当前版本只记录，不做 Agentic RL。

## 13. Eval 技术路线

### 13.1 Eval 目标

Eval 用于衡量 Harness 改造是否真的提高了 Agent 效果。

不是只看最终回答，而是分模块评估：

* 检索是否命中；
* 工具是否选对；
* 工具是否执行成功；
* 答案是否有依据；
* 任务是否完成；
* 延迟是否可接受。

### 13.2 测试集设计

构造 30-100 条测试任务，覆盖：

* 企业制度问答；
* 流程判断；
* 项目资料分析；
* 数据分析；
* 报告生成。

每条样本可以包含：

```json
{
  "query": "...",
  "user_role": "employee",
  "expected_task_type": "knowledge_qa",
  "expected_tools": ["search_kb"],
  "gold_docs": ["报销制度.md"],
  "expected_answer_points": ["材料清单", "审批流程"],
  "risk_level": "low"
}
```

### 13.3 指标设计

#### RAG Recall@K

```text
Recall@K = top-k 中命中 gold_doc 的样本数 / 有 gold_doc 标注的样本数
```

#### Citation Accuracy

```text
Citation Accuracy = 引用能支撑答案的样本数 / 带引用答案样本数
```

#### Tool Call Accuracy

```text
Tool Call Accuracy = 工具选择正确的样本数 / 需要工具调用的样本数
```

#### Tool Success Rate

```text
Tool Success Rate = 工具成功执行次数 / 工具调用总次数
```

#### Task Success Rate

```text
Task Success Rate = 任务完成样本数 / 总样本数
```

#### Verifier Pass Rate

```text
Verifier Pass Rate = 通过 Verifier 的样本数 / 总样本数
```

#### Average Latency

```text
Average Latency = 所有任务耗时总和 / 任务数
```

#### Error Type Distribution

统计：

* retrieval_failure；
* tool_error；
* permission_denied；
* missing_citation；
* format_error；
* unsupported_task；
* verifier_failed。

### 13.4 消融实验设计

可以对比：

```text
Baseline RAG
→ + Tool Use
→ + Tool Contract
→ + Verifier
→ + Retry
→ + Permission
→ + Trace Eval
```

评估每个模块带来的收益。

注意：

如果没有真实实验结果，不写具体提升数值，只写评估设计和计划指标。

## 16. 技术难点

### 16.1 Agent 执行链路稳定性

难点：

* LLM 可能选错工具；
* 工具参数可能不完整；
* SQL 可能执行失败；
* RAG 可能检索不到证据；
* 最终回答可能没有引用。

解决思路：

* Tool Contract；
* Router；
* Verifier；
* Retry；
* Trace 分析。

### 16.2 RAG 检索质量

难点：

* 用户 query 和制度文档措辞可能不一致；
* chunk 切分可能破坏语义；
* top-k 结果可能不相关；
* 多文档信息可能冲突。

解决思路：

* 合理 chunk；
* metadata；
* top-k；
* 可选 rerank；
* Citation Accuracy 评估。

### 16.3 工具调用可控性

难点：

* 工具调用容易失控；
* 参数容易错误；
* 权限边界需要明确；
* 高风险操作不能自动执行。

解决思路：

* input_schema；
* permission；
* risk_level；
* HITL；
* Trace Logging。

### 16.4 评估困难

难点：

* Agent 的效果不是单一指标；
* 失败原因可能来自检索、工具、规划或生成；
* 只看最终答案无法定位问题。

解决思路：

* 分模块评估；
* Trace 记录；
* Error Type Distribution；
* 消融实验。

## 18. 技术路线结论

本项目的技术路线核心不是“做一个简单 RAG 问答”，而是通过 Harness Runtime 将 RAG、工具调用、权限控制、结果校验和 Trace 评估组织成一个完整的 Agent 执行闭环。

当前版本应优先完成：

```text
RAG 检索
+ Tool Contract
+ Tool Use
+ Planner / Router
+ Runtime 主流程
+ Trace Logging
+ Eval Set
```

然后增强：

```text
Verifier
+ Retry
+ Permission / Safety
+ Human-in-the-loop 简化版
```

最终形成一个能够展示以下能力的 Demo：

1. 企业知识检索能力；
2. 业务工具调用能力；
3. 多步骤任务执行能力；
4. Agent Harness 工程化能力；
5. 结果校验与失败恢复能力；
6. 执行轨迹记录与量化评估能力。

该技术路线与第一个微调项目形成互补：

```text
微调项目：模型训练与对齐能力
Agent 项目：RAG、工具调用和应用工程化能力
```
